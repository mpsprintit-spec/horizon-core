# HC-002 Implementation Specification

Document ID: HC-002-IS  
Source Blueprint: HC-002 AI Cluster Blueprint  
Version: 1.0.0  
Status: Draft implementation specification  
Scope: Documentation only; no Go code, business logic, or HC-001 behavior changes.

## 1. Purpose

This Implementation Specification translates the approved HC-002 AI Cluster Blueprint into a repository-level implementation plan for Horizon Core. The blueprint remains the Single Source of Truth. This document defines where future HC-002 software components should live, which public interfaces they should expose, how modules should depend on each other, and the intended data and event flow.

HC-002 adds an AI cluster around real-time perception, sensor fusion, intelligent decision making, autonomous device behavior, HUD assistance, and continuous learning. The implementation must extend the existing HC-001 repository structure without changing completed HC-001 behavior.

## 3. Folder Structure Specification

The following folder structure is the target shape for HC-002 implementation. Folders are specified for future implementation only and must not be created until their implementation step begins.

### 3.1 AI Cluster Core

- Folder: `services/ai/`
- Package: `ai`
- Interface: cluster orchestration, module lifecycle, health, readiness, and AI workload dispatch contracts.
- Repository: no direct database repository in the core; delegate persistence to `services/storage/` or module-specific repositories.
- Service: `ClusterService`, `ModuleRegistryService`, `InferenceRoutingService`.
- Handler: cluster health, module status, and AI inference request handlers.
- gRPC: `AIClusterService` for internal service-to-service inference and module coordination.
- HTTP: `/ai/health`, `/ai/modules`, `/ai/infer`, `/ai/status`.
- Event: subscribes to input and context events; publishes AI processing and decision events.

### 3.2 Vision AI

- Folder: `services/ai/vision/`
- Package: `vision`
- Interface: `FrameAnalyzer`, `ObjectDetector`, `SceneUnderstander`, `DepthEstimator`, `OpticalFlowTracker`, `LowLightEnhancer`.
- Repository: `VisionResultRepository` for detection metadata and frame analysis references; raw media remains in Media or Storage services.
- Service: `VisionService` coordinates model inference and produces perception outputs.
- Handler: frame analysis command handler and media stream subscription handler.
- gRPC: `VisionAIService.AnalyzeFrame`, `VisionAIService.AnalyzeStream`, `VisionAIService.GetVisionResult`.
- HTTP: `/ai/vision/frames:analyze`, `/ai/vision/streams:analyze`, `/ai/vision/results/{id}`.
- Event: consumes `FrameCaptured`, `VideoFrameReceived`, `LowLightFrameReceived`; publishes `FrameAnalyzed`, `ObjectDetected`, `SceneUnderstood`, `DepthEstimated`, `OpticalFlowTracked`.

### 3.3 Sensor Fusion AI

- Folder: `services/ai/sensor-fusion/`
- Package: `sensorfusion`
- Interface: `SensorFusionEngine`, `IMUFuser`, `GPSFuser`, `VIOEstimator`, `NoiseFilter`, `StateEstimator`.
- Repository: `NavigationStateRepository` and `SensorSnapshotRepository` for fused state history.
- Service: `SensorFusionService` merges telemetry, IMU, GPS, visual features, and device state.
- Handler: sensor packet handler, telemetry subscription handler, and state estimation query handler.
- gRPC: `SensorFusionAIService.FuseSensors`, `SensorFusionAIService.EstimateState`, `SensorFusionAIService.GetNavigationState`.
- HTTP: `/ai/sensor-fusion:fuse`, `/ai/sensor-fusion/state`, `/ai/sensor-fusion/state/{missionId}`.
- Event: consumes `SensorDataReceived`, `TelemetryReceived`, `LocationUpdated`, `FrameAnalyzed`; publishes `SensorFused`, `NavigationStateEstimated`, `NoiseFiltered`, `ObstacleTracked`.

### 3.4 Decision AI

- Folder: `services/ai/decision/`
- Package: `decision`
- Interface: `SituationAssessor`, `RiskAnalyzer`, `GoalPlanner`, `BehaviorSelector`, `DecisionOptimizer`, `ContextAwareDecisionEngine`.
- Repository: `DecisionRepository`, `RiskAssessmentRepository`, and `GoalPlanRepository`.
- Service: `DecisionService` receives context and fused perception, then produces decisions and behavior intents.
- Handler: decision command handler, context update handler, and mission decision query handler.
- gRPC: `DecisionAIService.CreateDecision`, `DecisionAIService.AssessSituation`, `DecisionAIService.SelectBehavior`.
- HTTP: `/ai/decision:create`, `/ai/decision/assess`, `/ai/decision/behavior:select`, `/ai/decision/{id}`.
- Event: consumes `NavigationStateEstimated`, `ObjectDetected`, `ObstacleDetected`, `MissionContextUpdated`, `UserContextUpdated`, `SystemStatusUpdated`; publishes `SituationAssessed`, `RiskAnalyzed`, `GoalPlanCreated`, `BehaviorSelected`, `DecisionCreated`.

### 3.5 Learning Engine

- Folder: `services/ai/learning/`
- Package: `learning`
- Interface: `ExperienceRecorder`, `ModelTrainingCoordinator`, `UserPreferenceLearner`, `AnomalyLearner`, `PerformanceImprover`.
- Repository: `LearningRecordRepository`, `ExperienceMemoryRepository`, `ModelVersionRepository`, `UserPreferenceRepository`.
- Service: `LearningService` records outcomes, updates learning state, and coordinates model improvement workflows.
- Handler: learning event handler, feedback handler, model update handler.
- gRPC: `LearningEngineService.RecordExperience`, `LearningEngineService.ApplyFeedback`, `LearningEngineService.GetLearningState`.
- HTTP: `/ai/learning/experience`, `/ai/learning/feedback`, `/ai/learning/state/{missionId}`, `/ai/learning/models`.
- Event: consumes `DecisionCreated`, `BehaviorExecuted`, `DroneCommandExecuted`, `HUDRendered`, `MissionCompleted`, `UserFeedbackReceived`, `FailureLogged`, `EnvironmentalChangeDetected`; publishes `ExperienceRecorded`, `LearningUpdated`, `AnomalyLearned`, `ModelImprovementRequested`, `UserPreferenceLearned`.

### 3.6 Behavior Service

- Folder: `services/behavior/`
- Package: `main` initially; future implementation may introduce internal domain packages under this service.
- Interface: `BehaviorExecutor`, `BehaviorPolicy`, `BehaviorPlanRepository`, `ActionDispatcher`.
- Repository: `BehaviorPlanRepository` and `BehaviorExecutionRepository`.
- Service: `BehaviorService` turns Decision AI output into executable behavior plans.
- Handler: behavior selection event handler and behavior execution command handler.
- gRPC: `BehaviorService.ExecuteBehavior`, `BehaviorService.GetBehaviorPlan`.
- HTTP: `/behavior:execute`, `/behavior/plans/{id}`, `/behavior/executions/{id}`.
- Event: consumes `BehaviorSelected`, `DecisionCreated`; publishes `BehaviorPlanned`, `BehaviorExecuted`, `BehaviorFailed`.

### 3.7 Drone AI

- Folder: `services/drone/`
- Package: `main` initially; future implementation may introduce `drone`, `navigation`, and `control` packages inside the service.
- Interface: `PathPlanner`, `ObstacleAvoider`, `AutoTracker`, `FlightStabilizer`, `EnergyManager`, `DroneCommandDispatcher`.
- Repository: `FlightPlanRepository`, `DroneCommandRepository`, `DroneTelemetryRepository`.
- Service: `DroneAIService` creates and dispatches safe drone commands from behavior plans and navigation state.
- Handler: drone command handler, telemetry handler, obstacle update handler.
- gRPC: `DroneAIService.PlanPath`, `DroneAIService.DispatchCommand`, `DroneAIService.GetFlightState`.
- HTTP: `/drone/path:plan`, `/drone/commands`, `/drone/state/{droneId}`.
- Event: consumes `BehaviorExecuted`, `NavigationStateEstimated`, `ObstacleDetected`, `ObstacleTracked`, `BatteryStatusUpdated`; publishes `PathPlanned`, `ObstacleAvoidancePlanned`, `DroneCommandCreated`, `DroneCommandDispatched`, `DroneCommandExecuted`, `FlightStabilized`, `EnergyPlanUpdated`.

### 3.8 HUD AI / Glasses AI

- Folder: `services/glasses/`
- Package: `main` initially; future implementation may introduce `hud`, `ar`, and `attention` packages inside the service.
- Interface: `InformationPrioritizer`, `ARPlacementEngine`, `TargetHighlighter`, `AlertNotifier`, `AttentionManager`, `HUDRenderer`.
- Repository: `HUDSessionRepository`, `HUDDataRepository`, `AlertRepository`.
- Service: `HUDAIService` determines what information should be shown to the user and how it should be displayed.
- Handler: HUD update handler, alert handler, user attention handler.
- gRPC: `HUDAIService.RenderHUD`, `HUDAIService.PrioritizeInformation`, `HUDAIService.SendAlert`.
- HTTP: `/glasses/hud:render`, `/glasses/hud:data`, `/glasses/alerts`.
- Event: consumes `DecisionCreated`, `BehaviorSelected`, `ObjectDetected`, `TargetHighlighted`, `UserAttentionChanged`, `SystemAlertCreated`; publishes `HUDDataPrepared`, `HUDRendered`, `AlertPrioritized`, `AlertDisplayed`, `UserAttentionUpdated`.

## 4. Dependency Graph

### 4.1 Core Dependencies

- `services/ai/vision/` depends on media input, device metadata, and optional mission context. It does not depend on Decision AI, Drone AI, HUD AI, or Learning Engine.
- `services/ai/sensor-fusion/` depends on telemetry, sensors, location, device metadata, and selected Vision AI outputs. It does not depend on Decision AI.
- `services/ai/decision/` depends on Vision AI outputs, Sensor Fusion AI outputs, mission context, user context, system status, history, and rules or model metadata.
- `services/behavior/` depends on Decision AI output and exposes behavior execution state to Drone AI, HUD AI, and Learning Engine.
- `services/drone/` depends on Behavior Service, Sensor Fusion AI, device registry, radio connectivity, and mission context.
- `services/glasses/` depends on Decision AI, Behavior Service, Vision AI outputs, user context, and notification semantics.
- `services/ai/learning/` depends on events emitted by all other modules. It should not be a synchronous dependency for real-time decision paths.

### 4.2 Independent or Low-Coupled Modules

- Vision AI can run independently on frames and streams.
- Sensor Fusion AI can run independently on telemetry and sensor packets, but improves when Vision AI features are available.
- Learning Engine must be asynchronous from the real-time control path.
- Analytics, Storage, OTA, and Notification are platform services and should remain reusable by multiple AI modules.

### 4.3 Prohibited Dependency Direction

- Vision AI must not call Decision AI directly.
- Sensor Fusion AI must not call Decision AI directly for command selection.
- Learning Engine must not block Vision AI, Sensor Fusion AI, Decision AI, Drone AI, or HUD AI execution.
- Drone AI and HUD AI must not mutate Decision AI outputs; they may enrich execution state or render data derived from decisions.

## 5. Data Flow

1. Input
   - Sources: smart glasses, drone, smartwatch, smartphone or hub, external environment data, user commands, mission data, and system status.
   - Input objects: `Frame`, `AudioSample`, `SensorData`, `Telemetry`, `LocationSignal`, `UserContext`, `MissionContext`.

2. Vision
   - Consumes camera frames and video streams.
   - Produces detections, scene understanding, depth maps, optical flow, and vision confidence scores.

3. Sensor Fusion
   - Consumes IMU, GPS, telemetry, location, visual features, and device state.
   - Produces fused sensor state, navigation state, tracked obstacles, and confidence estimates.

4. Decision
   - Consumes fused state, vision outputs, mission data, user profile, environment data, system status, and history.
   - Produces assessed situations, risk analysis, goal plans, behavior selections, and decisions.

5. Behavior
   - Consumes decisions and behavior selections.
   - Produces behavior plans and execution requests for downstream services.

6. Drone / HUD
   - Drone AI consumes behavior plans, navigation state, obstacles, and battery state to create drone commands.
   - HUD AI consumes decisions, user attention state, detections, mission state, and alerts to create HUD data.

7. Learning
   - Consumes mission history, success and failure logs, user feedback, environmental changes, decisions, executions, drone commands, and HUD rendering results.
   - Produces learning records, preference updates, anomaly insights, performance improvements, and model improvement requests.

Canonical flow:

```text
Input
  -> Vision
  -> Sensor Fusion
  -> Decision
  -> Behavior
  -> Drone / HUD
  -> Learning
```

## 6. Public Interface Specification

Interfaces listed below define required contracts only. They must not be implemented in this specification.

### 6.1 Shared AI Interfaces

- `AICluster`: starts, stops, and reports module status.
- `AIModule`: standard lifecycle and health contract for AI modules.
- `InferenceRequestHandler`: accepts inference requests with context and returns typed AI results.
- `EventPublisher`: publishes domain events.
- `EventSubscriber`: subscribes to domain events.
- `ModelRegistry`: locates approved AI model versions and runtime metadata.
- `ContextProvider`: provides mission, user, environment, and system context.

### 6.2 Vision Interfaces

- `FrameAnalyzer`: analyzes one frame and returns a `VisionResult`.
- `StreamAnalyzer`: analyzes continuous video streams.
- `ObjectDetector`: returns `Detection` objects.
- `SceneUnderstander`: returns `SceneContext`.
- `DepthEstimator`: returns `DepthMap` or depth summary.
- `OpticalFlowTracker`: tracks motion between frames.
- `LowLightEnhancer`: improves low-light frame quality before inference.

### 6.3 Sensor Fusion Interfaces

- `SensorFusionEngine`: merges sensor samples into a fused state.
- `IMUFuser`: processes accelerometer, gyroscope, and magnetometer signals.
- `GPSFuser`: processes GPS and location data.
- `VIOEstimator`: combines visual and inertial odometry.
- `NoiseFilter`: filters unreliable sensor measurements.
- `StateEstimator`: creates `NavigationState` and confidence metrics.

### 6.4 Decision Interfaces

- `SituationAssessor`: interprets current mission and environment state.
- `RiskAnalyzer`: determines safety and operational risk.
- `GoalPlanner`: creates or updates mission goals.
- `BehaviorSelector`: chooses a behavior from assessed state and goals.
- `DecisionOptimizer`: performs multi-objective optimization.
- `ContextAwareDecisionEngine`: creates final `Decision` objects.

### 6.5 Learning Interfaces

- `ExperienceRecorder`: stores mission and action outcomes.
- `FeedbackProcessor`: processes user and system feedback.
- `ModelTrainingCoordinator`: schedules or requests model improvement.
- `UserPreferenceLearner`: updates learned user preferences.
- `AnomalyLearner`: converts anomalies into learning records.
- `PerformanceImprover`: evaluates model and behavior performance.

### 6.6 Behavior Interfaces

- `BehaviorExecutor`: executes a selected behavior.
- `BehaviorPlanner`: converts decisions into behavior plans.
- `ActionDispatcher`: dispatches actions to device-specific services.
- `BehaviorPolicy`: validates behavior safety and policy compliance.

### 6.7 Drone Interfaces

- `PathPlanner`: creates safe flight paths.
- `ObstacleAvoider`: adjusts paths around obstacles.
- `AutoTracker`: tracks user, object, or mission target.
- `FlightStabilizer`: stabilizes flight commands.
- `EnergyManager`: optimizes energy usage and return-to-base behavior.
- `DroneCommandDispatcher`: dispatches commands to drones.

### 6.8 HUD Interfaces

- `InformationPrioritizer`: ranks information for display.
- `ARPlacementEngine`: places augmented content in the user's field of view.
- `TargetHighlighter`: marks relevant targets.
- `AlertNotifier`: prepares attention-worthy alerts.
- `AttentionManager`: tracks user focus and avoids overload.
- `HUDRenderer`: renders final `HUDData`.

## 9. Implementation Roadmap

### Step 1: Shared AI Contracts and Documentation

- Define shared HC-002 domain contracts under the AI service boundary.
- Establish event naming, module lifecycle conventions, and transport contract strategy.
- Keep implementation contract-only until module boundaries are approved.

### Step 2: Vision AI

- Add Vision AI module structure.
- Define frame and stream analysis contracts.
- Integrate with media input events.
- Emit perception and detection events.

### Step 3: Sensor Fusion AI

- Add Sensor Fusion AI module structure.
- Define sensor, telemetry, location, and visual feature fusion contracts.
- Produce navigation state and obstacle tracking outputs.

### Step 4: Decision AI

- Add Decision AI module structure.
- Define situation assessment, risk analysis, goal planning, behavior selection, and decision contracts.
- Subscribe to fused state, mission context, user context, and perception events.

### Step 5: Behavior Service Integration

- Extend Behavior service contracts for selected behaviors and behavior plans.
- Translate decisions into executable behavior plans.
- Publish behavior execution events for Drone AI, HUD AI, and Learning Engine.

### Step 6: Drone AI

- Extend Drone service contracts for autonomous flight intelligence.
- Implement path planning, obstacle avoidance, tracking, stabilization, and energy management boundaries.
- Dispatch validated drone commands only after behavior and safety checks.

### Step 7: HUD AI / Glasses AI

- Extend Glasses service contracts for HUD intelligence.
- Implement information prioritization, AR placement, target highlighting, alerts, and attention management boundaries.
- Ensure HUD output remains user-centered and does not block flight control paths.

### Step 8: Learning Engine

- Add Learning Engine module structure.
- Record experience, mission outcomes, feedback, failures, environmental changes, and model improvement signals.
- Keep learning asynchronous from real-time perception, decision, drone, and HUD execution.

### Step 9: Platform Integrations

- Connect Storage for learning records, decisions, sensor snapshots, and event history.
- Connect Analytics for performance and mission reporting.
- Connect OTA for model and edge runtime delivery.
- Connect Notification for alerts generated by Decision AI and HUD AI.

### Step 10: Deployment Profiles

- Define cloud AI profile for heavy training and global intelligence.
- Define edge AI profile for on-device inference and low latency.
- Define hybrid intelligence profile combining cloud and edge outputs.

### Step 11: Verification and Acceptance

- Verify that each module follows the approved dependency direction.
- Verify that real-time control paths do not synchronously depend on Learning Engine.
- Verify that all domain events have one publisher and clear subscribers.
- Verify that public interfaces remain implementation-free until corresponding implementation work begins.

## 10. Non-Goals for This Specification

- No Go files are created or modified.
- No business logic is implemented.
- No repository, database, queue, or infrastructure resource is created.
- No HC-001 behavior is changed.
- No AI model is selected, trained, deployed, or configured.
